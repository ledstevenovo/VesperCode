# 1. 问题陈述

## 1.1 项目定义与背景

**项目名称：** VesperCode

**一句话定位：**

面向维护 Windows 本地代码仓库的独立开发者，构建一个以治理为核心、具备确定性护栏和测试反馈闭环的 Coding Agent Harness，使 LLM 能在受控工作区中检查和修改代码、运行验证并根据反馈自我修正，同时显式管理路径、动作、审批、记忆、配置和数据披露风险。

普通聊天模型只能生成建议或代码片段，不能可靠地读取真实仓库、应用补丁、运行测试并根据结果持续修正。即使为其提供工具，仅依靠系统提示也无法确定性地阻止路径越界、危险动作、秘密读取、验收条件篡改或未经批准的持久修改。VesperCode 因此自行实现 Agent 主循环、工具协议、治理机制、反馈闭环、记忆和配置，而不是把安全要求交给模型自觉遵守。

## 1.2 目标用户、支持范围与产品交互面

目标用户是维护 Windows 本地代码仓库、希望在人工监督下自动修复缺陷的独立开发者。“Windows 本地代码仓库”描述宿主运行环境，不表示 Harness 只能处理 Windows 程序。

核心决策循环、工具协议、记忆、治理、反馈和配置机制保持语言无关。首版提供面向明确兼容性画像的 Python 项目适配器，不承诺覆盖完整 Python 生态。该适配器支持通过 pytest 收集和运行测试，并运行项目已配置的 Ruff 和 Mypy 检查。`unittest` 测试仅在 pytest 能直接收集时受支持。

VesperCode 不负责创建开发环境或安装项目依赖。执行环境必须预先准备完成。VesperCode 自身使用的 Python 版本与目标项目支持的 Python 版本分别定义。目标 Python 版本、项目布局、配置文件和环境入口将在技术选型章节形成可测试的支持矩阵。画像之外的项目以 `UNSUPPORTED_PROJECT` 停止，不得降级为任意命令执行。

首版只接受具有有效 `HEAD` 的 Git 工作区。已暂存修改、未暂存修改或非忽略的未跟踪文件都会使运行以 `WORKTREE_DIRTY` 停止。VesperCode 不自动执行 commit、stash、clean 或删除操作。Git 忽略文件不进入工作区快照、不复制到执行副本，检查也不得依赖这些文件。

VesperCode 以同一 Harness 内核提供两种部署配置：

- **本地实际使用模式：** 在用户 Windows 本机运行，通过 CLI 启动仅绑定 `127.0.0.1` 的本地 WebUI，用于指定工作区、启动修复、查看候选 diff、处理审批、观察结构化检查结果和查阅审计记录。本地服务使用本地会话令牌、严格的 Origin/Host 校验和 CSRF 防护，不绑定公网接口。
- **公网演示模式：** 使用内置示例仓库、预定义演示场景、Mock LLM 和受限 `DemoExecutor`，展示路径拦截、审批、反馈回灌、补丁验证和审计流程。演示模式不访问用户本地文件，不允许上传任意仓库，不接收真实 LLM 凭据，不启用 Native 执行器，不执行任意命令或对外网络请求，并在每次演示后重置可丢弃状态。

两种模式尽量复用同一前端和 API 状态模型。具体前端技术、Open Design 设计系统与 UI skill 在技术选型章节说明。

## 1.3 要解决的核心问题

VesperCode 需要解决以下问题：

1. 自行实现可注入 Mock LLM 的决策循环，使 Agent 能执行“观察—决策—动作—反馈—修正—停止”，而不使用替代主循环的高层 Agent 框架。
2. 通过受治理的文件读取、补丁语义解析和补丁应用工具处理仓库内容。LLM 输出首先被视为内存中的不可信候选补丁。系统对补丁来源路径和目标路径执行语义解析、Windows 路径规范化、重解析点检查、边界验证和前置摘要验证。候选补丁只能应用到指定 `ExecutionWorkspace`；只有经过全新执行副本验证、绑定精确摘要并取得一次性批准的最终补丁，才能由 `PatchApplyTool` 持久化到 `AuthoritativeWorkspace`。
3. 对所有规范化结构化动作进行风险分类，产生 `ALLOW / ASK / DENY` 决策。检查执行请求由适配器生成，至少包含允许的可执行程序、独立 argv、工作目录、超时和输出上限，并且不经 Shell 解释。首版不接受 PowerShell、cmd、Shell 脚本或任意命令文本。
4. 实现不可复用的一次性审批状态机。批准必须绑定完整动作上下文、具有明确有效期并被原子消费；硬 `DENY` 规则不能被批准覆盖。首版不提供有限会话授权。
5. 将测试、Lint 和类型检查转换为退出码与结构化结果，并把有效反馈回灌给 LLM，而不是让模型仅凭文本猜测是否成功。
6. 按需向 LLM 提供项目约定、历史决策、上次运行摘要和已知失败，不加载完整历史上下文。
7. 通过声明式配置定义工作区、工具权限、检查适配器、执行器、运行预算和日志策略。未知字段、无效值、试图覆盖硬 `DENY` 规则，以及通过默认值或隐式行为扩大权限的配置，必须确定性报错并失败关闭。Native 等受支持的高风险模式只能通过显式配置启动新的运行，并强制显示警告、逐次审批和审计标记；运行过程中不得切换执行器。
8. 在基线阶段生成并封存不可变、版本化的 `ValidationManifest`。Manifest 记录目标测试 ID、完整测试收集结果、结构化检查动作、工具版本、执行器标识、检查配置文件摘要、受保护测试文件摘要及依赖清单摘要，并以规范化内容和哈希存储在 Agent 无法访问的运行数据库中。
9. 将环境准备与 Agent 循环分离，不在运行过程中自动联网安装依赖、拉取镜像或构建环境。

### 工作区与补丁事务

`AuthoritativeWorkspace` 是用户的权威工作区；`ExecutionWorkspace` 是从确定快照创建的可丢弃执行副本。运行采用以下顺序：

1. 记录权威工作区的 Git 提交、文件集合和内容摘要。
2. 从该快照创建执行副本。LLM 后续只读取执行副本并生成内存候选补丁。
3. `PatchApplyTool` 将候选补丁应用到执行副本，Agent 在其中迭代和验证。
4. 得到候选结果后，系统计算规范化最终 diff 和摘要，并从原始快照创建全新的执行副本。
5. 系统在新副本中应用完全相同的最终 diff，完成目标测试、全量测试和所有已配置检查。
6. 系统向用户展示精确最终 diff、验证结果和 Manifest，取得绑定完整上下文的一次性持久化批准。
7. 持久化前再次确认权威工作区仍与原始快照一致；发生变化时以 `WORKSPACE_CHANGED` 停止，禁止覆盖用户的新修改。
8. `PatchApplyTool` 将同一补丁持久化，并验证最终文件集合和摘要与已验证执行副本一致。补丁工具可以利用写入日志恢复自身的部分写入失败，但不采用“持久化后测试失败再自动回滚”的流程。

### 默认动作决策

| 动作 | 默认决策 | 约束 |
| --- | --- | --- |
| 列出或读取执行副本中的普通文件 | `ALLOW` | 必须通过路径围栏 |
| 读取匹配硬拒绝规则的敏感路径 | `DENY` | 不承诺自动识别所有凭据 |
| 读取工作区外文件 | `DENY` | 不可通过批准覆盖 |
| 在执行副本应用普通候选补丁 | `ALLOW` | 不得触及验收契约 |
| 应用生成的复现测试补丁 | `ASK` | 绑定测试内容、目标 ID、动作和失败匹配器 |
| 在默认 Docker 中运行结构化检查 | `ALLOW` | 仅限适配器动作，默认禁用外部网络 |
| 在 Native 中执行任何检查 | `ASK` | 每次单独批准，不允许会话授权 |
| 向权威工作区持久化最终 diff | `ASK` | 两个使用场景均适用 |
| 修改验收契约 | `DENY` | 包括测试、配置、检查动作和 Manifest |
| 通用联网、下载、发布或推送 | `DENY` | LLM 控制面通信除外 |
| 绕过执行器或执行任意 Shell | `DENY` | 包括明确的破坏性系统动作 |

“联网 `DENY`”约束 Agent 工具和可识别的结构化动作。Native 子进程内部间接联网仍无法在操作系统层面保证阻断。

每次 `ASK` 生成不可复用的 `ApprovalContext`，至少绑定 `run_id`、动作类型、执行器类型、规范化参数或 diff 摘要、权威工作区标识、工作区快照摘要、`ValidationManifest` 摘要、Docker 镜像摘要、网络策略和 `expires_at`。批准的剩余使用次数固定为 1，并通过 SQLite 事务原子消费。任何字段变化、过期或重复使用都必须重新判定。

### 验收契约防篡改

进入修复阶段后，已有测试、已确认的复现测试、测试收集配置、Ruff/Mypy 配置、结构化检查动作和依赖清单不得由修复补丁修改。补丁直接触及受保护内容时必须在应用前被拒绝；验证前发现内容漂移时以 `VALIDATION_TAMPERED` 停止。工具版本、执行镜像或执行环境发生变化时以 `VALIDATION_ENV_CHANGED` 停止。

已有失败测试场景在基线后封存 `ValidationManifest v1`。自然语言场景先封存原始基线，在复现测试获批并被稳定复现后派生不可变的 `ValidationManifest v2`；两个版本均保留，修复阶段以 `v2` 为当前验收契约。合法改变验收条件必须终止当前运行并重新建立基线，不能在运行中重新授权弱化契约。

## 1.4 核心使用场景

### 场景一：根据已有失败测试修复缺陷

用户指定工作区和目标测试集合。VesperCode 在从同一权威快照创建的全新执行副本中建立基线：

- 所有目标测试 ID 必须成功收集；任何目标未收集时以 `TARGET_NOT_FOUND` 停止。
- 目标集合中至少一个测试必须在测试调用阶段产生确定的 `FAIL`。目标全部通过，或仅产生 `SKIPPED`、`XFAIL`、收集错误、环境错误或超时时，以 `TARGET_NOT_REPRODUCED` 停止。
- 目标集合之外的 pytest 测试必须通过，所有已配置的 Ruff 和 Mypy 检查也必须通过。代码检查得到 `FAIL` 时以 `BASELINE_BLOCKED` 停止；得到 `ERROR` 或 `TIMEOUT` 时保留对应状态并停止。
- 系统在第二个全新执行副本中重新运行首次失败的目标测试。两次运行必须产生相同的规范化失败指纹，否则以 `TARGET_UNSTABLE` 停止。

基线确认后，系统封存目标失败指纹和 `ValidationManifest v1`。Agent 使用受治理的工具检查代码、生成候选补丁并在执行副本中运行目标测试。失败结果被结构化后回灌给 LLM，Agent 可以继续修正，直到所有成功条件满足或触发停止条件。目标集合中原本通过的测试也保留在最终验收契约中。

### 场景二：根据自然语言缺陷生成复现测试

当用户只提供缺陷描述时，原始项目的全量 pytest 和所有已配置的 Ruff、Mypy 检查必须首先通过，否则不得生成修复。

Agent 在内存中生成只新增复现测试、不修改已有测试或检查配置的测试补丁，并向用户展示测试内容、目标测试 ID、结构化执行动作和预期失败匹配器。用户批准后，系统从同一基线分别创建两个全新执行副本，在每个副本中应用相同测试补丁并各运行一次。复现确认前，测试补丁不得写入权威工作区。

目标测试必须在两个副本中被实际收集和执行，两次结果必须产生相同且匹配批准内容的规范化失败指纹。语法错误、测试收集失败、依赖缺失、导入失败、setup/teardown 错误、环境错误、未执行到目标测试、无关异常和超时均不构成有效复现。测试补丁不得导致目标之外的测试失败或已配置的 Ruff/Mypy 检查失败。无法得到有效复现时，系统以 `REPRO_NOT_CONFIRMED` 停止并报告执行证据。

复现确认后，系统生成包含测试内容、目标 ID、补丁摘要和失败指纹的 `ValidationManifest v2`。该测试在修复阶段不得修改或删除，最终必须与业务代码修复组成同一个精确 diff，在全新执行副本中完成验证，并在一次性批准后共同持久化。

## 1.5 成功定义、停止条件与主要贡献

单次修复只有在以下条件全部满足时才能标记成功：

- Manifest 中的所有目标测试 ID 均以原 ID 被实际收集、执行并报告为 `PASS`；`SKIPPED`、`XFAIL`、`XPASS`、`DESELECTED` 和未收集均不构成通过。
- 全量 pytest 为 `PASS`。
- 所有已配置的 Ruff 和 Mypy 检查为 `PASS`。
- 最终测试集合、检查动作、受保护文件及配置摘要与当前封存的 `ValidationManifest` 一致。
- 自然语言场景中，获批的复现测试以相同内容保留在最终 diff 中并实际通过。
- 最终精确 diff 已在从原始快照重新创建的执行副本中完成验证。
- 用户已批准绑定完整上下文的持久化动作，最终补丁已由 `PatchApplyTool` 写入权威工作区，且权威文件摘要与已验证候选状态一致。
- 没有未解决的审批、超时、执行错误、基线阻断或环境漂移状态。

检查状态至少包括：

- `NOT_CONFIGURED`：可选检查没有配置，不参与成功判定；
- `NOT_RUN`：已配置但尚未执行；
- `PASS`：检查得到确定的通过结果；
- `FAIL`：检查得到确定的失败结果；
- `TIMEOUT`：超过执行预算；
- `ERROR`：发生收集、环境、工具或结果解析错误。

已配置的检查只有 `PASS` 才能满足成功条件。已配置但工具缺失属于 `ERROR`，不能视为 `NOT_CONFIGURED`。

当达到配置规定的重复失败阈值、重复动作阈值、最大决策轮次、模型调用次数、输入输出 Token 预算、总运行时间、单动作超时或输出体量上限时，系统必须停止。预算类限制返回 `BUDGET_EXHAUSTED` 并记录具体原因；重复失败或动作达到阈值时返回 `NO_PROGRESS`。LLM 可以提出正常停止，但不能覆盖或忽略 Harness 预算。

当本次运行选择 `DockerExecutor` 时，Docker 不可用、镜像缺失或环境画像不匹配必须在任何权威工作区持久写入之前被预检发现，并使该次运行失败关闭；不得自动切换到 Native 模式。Native 模式必须由用户通过显式配置另行启动。

VesperCode 以治理为主要贡献，重点实现工作区围栏、结构化动作风险分类、一次性 HITL 审批状态机以及 `ValidationManifest` 验收契约防篡改。决策、工具、记忆、治理、反馈和配置六个 Harness 维度均提供最低可运行实现。即使使用 Mock LLM，也必须能够确定性测试路径围栏、动作分类、审批状态机、工具分发、记忆读写、配置校验、反馈回灌、验收契约和停止条件。

四类领域机制、六个 Harness 维度的编码接口、确定性测试与 Mock LLM 机制演示将在独立的“领域与机制设计”章节中详细规定。机制演示至少覆盖：危险动作在执行前被拦截；注入失败后 Mock LLM 的下一步动作发生确定变化；修改受保护测试或检查配置时返回 `DENY` 或 `VALIDATION_TAMPERED`。

## 1.6 安全承诺与非目标

VesperCode 保证其内建文件工具在经过 Windows 路径规范化、重解析点检查和最终边界验证后，只操作授权工作区。该承诺适用于 VesperCode 自身实现的工具，不等同于对任意子进程提供操作系统级文件隔离。

默认 `DockerExecutor` 仅挂载可丢弃执行副本，不挂载权威工作区、Docker 控制接口、宿主凭据目录或任意额外宿主路径；不使用 privileged 模式；不自动继承宿主环境变量，只注入显式白名单值；执行镜像按不可变 digest 绑定，并采用外部网络禁用、资源限制和独立临时写入区。具体参数在安全设计章节规定。Docker 只为受支持的容器兼容项目提供纵深防御，不构成适用于任意 Windows 程序的通用操作系统级沙箱。

显式启用的 `NativeExecutor` 继承当前 Windows 用户的文件和网络权限，因此不承诺已批准子进程无法访问工作区外文件或网络。Native 中每次检查都必须单独批准，且不能通过会话级授权绕过逐次确认。

VesperCode 控制面只与用户显式配置的 LLM 供应商通信。该通信不属于 Agent 的仓库工具权限；模型不能自行选择供应商地址或读取凭据。

凭据威胁模型规定 LLM Key 的安全录入、Keyring 存储、状态查看、更新、清除和日志隔离。LLM 数据披露模型另行规定允许发送的代码片段、文件内容、运行结果和记忆，并实施拒绝路径过滤、确定性脱敏、体量限制和披露范围审计。首版不承诺识别所有秘密格式。

持久记忆和运行时审计默认只保存结构化决策、动作摘要、文件摘要、审批结果、规范化失败指纹以及截断、脱敏后的检查输出。不默认持久化完整文件内容、原始提示、完整命令输出或凭据。完整输出只可在当前运行的受限缓冲区中短暂存在；用户能够检查并清除跨会话记忆。

首版明确不做：

- 多 Agent 协作；
- IDE 插件；
- GitHub、GitLab 等远程代码托管平台 API 集成；
- 通过公网服务访问或控制用户本地仓库的远程 Agent；
- 在公网演示模式中接收用户代码、仓库地址或真实凭据；
